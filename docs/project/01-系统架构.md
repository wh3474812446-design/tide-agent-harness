# 系统架构

## 架构概览

Tide 由七个核心层组成：

```text
用户界面 / CLI
  -> Tide Runtime
  -> Agent Loop
  -> Model Provider
  -> Tool Registry / Tool Executor
  -> Risk Policy
  -> Context Manager / Session Store
```

每一层职责独立，方便替换模型、添加工具、调整权限策略或扩展界面。

## 模块职责

| 模块 | 路径 | 职责 |
|---|---|---|
| Web Server | `src/server.ts` | 提供浏览器界面、聊天 API、模型配置 API |
| Runtime | `src/app/runtime.ts` | 组装 Provider、工具、权限、上下文和会话 |
| Agent Loop | `src/core/agent-loop.ts` | 执行模型请求、工具调用回填和最终回复 |
| Model Provider | `src/model/` | 将 Tide 内部消息转换为模型 API 请求 |
| Tool Registry | `src/tools/tool.ts` | 注册工具并生成模型可见的工具定义 |
| Tool Executor | `src/tools/executor.ts` | 校验工具参数、执行权限判断、运行工具 |
| Risk Policy | `src/policy/policy.ts` | 根据风险等级决定工具是否允许执行 |
| Context Manager | `src/context/context-manager.ts` | 控制上下文长度并保留必要工具配对 |
| Session Store | `src/session/session-store.ts` | 持久化会话消息 |

## 运行时组装

`createTideRuntime()` 负责读取环境变量并创建完整运行时：

- 加载模型配置。
- 创建模型 Provider。
- 注册内置工具。
- 加载 JSON API 工具。
- 创建权限策略。
- 创建上下文管理器和会话存储。

浏览器界面和 CLI 都依赖同一套 Runtime，因此工具行为和权限判断保持一致。

## 模型 Provider

Tide 的 Provider 接口屏蔽不同模型服务差异。当前支持：

- OpenAI-compatible：DeepSeek、Qwen、GLM、MiniMax、Kimi、小米 MiMo、自定义兼容端点。
- Anthropic：用于 Claude API。
- Scripted Provider：用于测试和演示。

Provider 输出统一的 `text`、`tool_call`、`tool_result` 消息块，Agent Loop 不需要关心底层 API 的细节。

## 工具执行链路

工具执行必须经过三层控制：

1. Registry 确认工具存在。
2. JSON Schema 校验输入参数。
3. Risk Policy 检查风险等级是否允许。

工具执行结果会被截断到安全长度，再作为 `tool_result` 回填给模型。

## 数据持久化

当前本地持久化包括：

- `.env`：模型、API Key、权限和工具配置。
- `.sessions/`：会话历史。
- `examples/api-tools.example.json`：API 工具配置示例。

未来可以把会话、审计、工具执行记录迁移到 SQLite 或服务端数据库。
