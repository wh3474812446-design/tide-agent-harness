# 扩展与集成架构

## 扩展目标

Tide 的扩展方向是让更多外部能力以受控工具形式接入，而不是把所有逻辑写死在 Agent Loop 中。

主要扩展点：

- 模型 Provider。
- 本地工具。
- HTTP API 工具。
- Web UI 面板。
- 会话和审计存储。
- 后续 MCP 或插件协议。

## 新增模型 Provider

新增模型服务时，应优先判断它是否兼容 OpenAI Chat Completions。

如果兼容，只需要新增配置项：

- API Key 环境变量。
- Base URL。
- Model 名称。
- Web UI 预设。

如果不兼容，则实现新的 Provider，并把模型消息转换为 Tide 内部消息块。

## 新增本地工具

新增本地工具应遵守：

1. 工具名使用稳定英文标识。
2. `description` 明确说明适用场景。
3. `inputSchema` 严格限制参数。
4. 正确声明 `risk`。
5. 写入类工具默认 `concurrencySafe: false`。
6. 本地路径必须经过工作区边界校验。
7. 添加自动化测试。

## HTTP API 工具

业务 API 建议优先通过 JSON 配置接入，而不是写 TypeScript 工具。

适合 JSON API 工具的场景：

- 查询类接口。
- 稳定的 REST API。
- 简单的 POST/GET 调用。
- 使用 API Key 或 Bearer Token 认证。

需要写本地工具的场景：

- 复杂签名。
- 多步认证。
- 文件上传下载。
- 长轮询或流式任务。
- 需要特殊错误恢复逻辑。

## Web UI 扩展

当前 Web UI 已提供：

- 聊天区。
- 模型配置。
- 工具列表。
- 运行事件。
- 系统状态。

可继续扩展：

- 文件管理面板。
- API 工具编辑器。
- 会话列表。
- 权限开关。
- 工具执行详情。
- 知识库管理。

## MCP 与插件预留

当前 Tide 未直接实现 MCP Host，但架构上已具备接入条件：

- Registry 可以注册来自外部协议的工具。
- Executor 可以统一处理 Schema 校验和权限策略。
- Event Bus 可以记录外部工具执行过程。
- Runtime 可以在启动阶段加载插件或远程工具。

后续接入 MCP 时，应先解决鉴权、工具命名冲突、输出截断和权限映射。
