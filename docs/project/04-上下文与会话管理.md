# 上下文与会话管理

## 目标

Tide 的上下文管理需要同时满足三件事：

- 控制发送给模型的消息长度。
- 保留最近对话和关键工具结果。
- 确保工具调用和工具结果成对出现。

这可以降低模型 API 成本，也能避免因为上下文裁剪破坏工具协议。

## 消息结构

Tide 内部消息由角色和内容块组成：

```ts
interface Message {
  role: "user" | "assistant";
  content: ContentBlock[];
}
```

常见内容块：

- `text`
- `tool_call`
- `tool_result`

模型 Provider 会把这些内部结构转换为各模型服务需要的请求格式。

## 上下文裁剪

当消息超过限制时，Context Manager 会压缩较早内容，只保留对当前任务更重要的尾部消息。

关键规则：

- 最新用户请求优先保留。
- 最近模型回复优先保留。
- 如果保留了 `tool_call`，对应 `tool_result` 也必须保留。
- 如果保留了 `tool_result`，对应 `tool_call` 也必须保留。

## 会话存储

会话保存在本地 `.sessions/` 目录中。每次运行结束后，Tide 会把会话消息写入本地文件。

当前适合：

- 本地调试。
- 单用户使用。
- 浏览器会话恢复。

生产环境可迁移到：

- SQLite。
- PostgreSQL。
- 对象存储。
- 带权限控制的企业数据层。

## 隐私和密钥

会话文件可能包含用户输入、工具输出和模型回复。需要注意：

- 不要把 `.sessions/` 直接提交到公开仓库。
- 不要让模型读取无关密钥文件。
- `.env` 只保存本地密钥，不应进入版本控制。
- 导出会话前应检查是否包含敏感信息。

## 后续增强

- 会话列表和搜索。
- 会话删除与归档。
- 长期记忆分层存储。
- 工具结果单独落盘，模型只接收摘要。
- 面向知识库的文档索引。
